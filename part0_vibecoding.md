## 1. Что такое вайбкодинг

### 1.1. Определение

Термин «vibe coding» (вайбкодинг) ввёл Андрей Карпатый — один из ведущих исследователей в области AI — в начале 2025 года.

Суть простая: вы описываете на обычном языке, что хотите получить, а AI создаёт за вас весь программный код. Если результат выглядит и работает нормально — принимаете. Если что-то не так — говорите AI, что не так, и он исправляет.

> «Забудь, что код существует. Просто опиши, что ты хочешь. Если что-то сломается — скажи AI, что сломалось. Вот и всё.»

Никакого знания программирования не требуется. Вы общаетесь с AI так же, как общались бы с коллегой-разработчиком.

### 1.2. Почему это взлетело

- **Порог входа упал до нуля** — не нужно уметь программировать, знать языки или инструменты
- **Скорость**: от идеи до работающего прототипа за минуты, а не дни или недели
- **Доступность**: всё работает прямо в браузере — ничего не нужно устанавливать
- **Результат виден сразу** — вы описали, AI сделал, вы видите работающее приложение

### 1.3. Кто этим пользуется

Вайбкодинг — это не только для программистов. Им уже пользуются:

- **Стартапы** — быстрая проверка идеи перед тем, как вкладывать деньги в полноценную разработку
- **Менеджеры и продакты** — создание интерактивных демо для клиентов и руководства, без привлечения разработчиков
- **Дизайнеры** — интерактивные прототипы вместо статичных картинок
- **Аналитики** — дашборды и визуализации данных за минуты
- **Разработчики** — быстрые внутренние утилиты, скрипты, эксперименты с новыми технологиями
- **Участники хакатонов** — максимальный результат за минимальное время

## 2. Инструменты вайбкодинга

### 2.1. Браузерные: открыл и работаешь

Самый простой вариант — не нужно ничего устанавливать. Вы открываете сайт, описываете что хотите, и получаете готовое приложение.

| Инструмент | Что делает | Сильная сторона |
|---|---|---|
| **Bolt** (bolt.new) | Создаёт полноценные веб-приложения прямо в браузере | Можно сразу опубликовать в интернете |
| **v0** (от Vercel) | Создаёт красивые интерфейсы и компоненты | Отличный дизайн «из коробки» |
| **Lovable** | Создаёт приложения с базой данных и авторизацией | Всё включено: от интерфейса до хранения данных |
| **Replit Agent** | Создаёт любые приложения в облаке | Работает с разными языками, сразу можно поделиться |

### 2.2. Встроенные в редактор кода: для тех, кто уже программирует

Эти инструменты работают внутри редактора кода и «видят» ваш существующий проект.

| Инструмент | Что делает | Сильная сторона |
|---|---|---|
| **Cursor Composer** | Генерирует код по описанию внутри редактора | Понимает контекст всего вашего проекта |
| **Windsurf Cascade** | Создаёт код в несколько шагов | Автоматически разбивает задачу на подзадачи |
| **GitHub Copilot Chat** | Генерирует код по описанию | Глубокая интеграция с GitHub |

### 2.3. Что выбрать

```
Нужен прототип за 10 минут?          → Bolt / Lovable
Нужен красивый интерфейс?            → v0
Работаете с существующим проектом?   → Cursor Composer
Хотите максимальный контроль?        → Cursor / Windsurf
Хакатон / демо для инвестора?        → Bolt + Lovable
```

## 3. Живой пример: от описания до приложения

### 3.1. Задача

Представьте, что вам нужен инструмент для анализа данных из таблиц. Вы пишете AI:

> «Сделай дашборд для анализа CSV-файлов: загрузка файла, таблица данных, фильтрация, три графика (столбчатый, линейный, круговой), тёмная тема.»

*(CSV — это обычный формат таблиц, который умеет сохранять Excel)*

### 3.2. Что происходит

**Один запрос → 30 секунд → работающее приложение.**

AI сам создаёт:
- Страницу с возможностью загрузить файл (перетащить мышкой)
- Таблицу с данными, которую можно сортировать и фильтровать
- Три графика, которые автоматически строятся по данным
- Приятный тёмный дизайн

### 3.3. Что получилось

Результат впечатляет:
- Рабочий интерфейс, которым сразу можно пользоваться
- Загрузка файла работает
- Графики отрисовываются
- Фильтрация функционирует
- Выглядит профессионально

### 3.4. Итерации: постепенное улучшение

Вайбкодинг — это не один запрос. Это диалог. Вы описываете, AI делает, вы просите доработать:

```
Запрос 1: "Сделай дашборд для CSV"
→ Появился базовый интерфейс с графиками

Запрос 2: "Добавь возможность фильтровать по колонкам"
→ Фильтры появились

Запрос 3: "Сделай тёмную тему"
→ Тема переключается

Запрос 4: "При загрузке большого файла всё тормозит"
→ AI оптимизирует работу с большими таблицами

Запрос 5: "Сохрани выбранные фильтры в ссылку, чтобы можно было поделиться"
→ Теперь ссылка содержит настройки фильтров
```

Каждая итерация — 10–30 секунд. Всё вместе — около 5 минут.

## 4. Где вайбкодинг работает отлично

### 4.1. Прототипы и демо

Главный сценарий: проверить идею до того, как тратить недели на полноценную разработку.

- Демо для инвестора, клиента или руководства
- Проверка гипотезы: «а будут ли люди этим пользоваться?»
- Наглядное ТЗ: вместо длинного документа — работающее приложение

> *«Мы за вечер собрали прототип, который раньше занял бы двухнедельный спринт. Инвестор увидел работающий продукт, а не слайды.»*

### 4.2. Внутренние инструменты команды

Те штуки, которые нужны, но на которые жалко тратить время разработчиков:

- Дашборды для отслеживания метрик и KPI
- Калькуляторы и конвертеры
- Формы для сбора данных (опросы, заявки, чеклисты)
- Визуализация отчётов
- Генераторы шаблонных документов

### 4.3. Одноразовые задачи

- Преобразовать данные из одного формата в другой
- Массово переименовать файлы
- Собрать разовый отчёт из нескольких источников
- Автоматизировать рутинную задачу, которая возникла один раз

### 4.4. Обучение и эксперименты

- Быстро попробовать новую технологию или сервис
- Создать наглядный пример для обучения
- Разобраться, как работает какой-то инструмент, увидев результат

### 4.5. Маркетинг и лендинги

- Посадочная страница для нового продукта за час
- Несколько вариантов страницы для тестирования (какой лучше конвертирует)
- Интерактивная демо-страница продукта

## 5. Вайбкодинг и другие способы работы с AI

### 5.1. Три подхода

Вайбкодинг — это один из способов использования AI в разработке. Есть и другие, более контролируемые:

```
Вайбкодинг          →    Агентский режим    →    Мультиагенты
     |                         |                       |
"Сделай мне"           "Сделай по правилам,     "Спланируй задачу,
                        я проверю каждый шаг"    распредели между
                                                  несколькими AI,
                                                  проверяй результат"
```

- **Вайбкодинг** — описываете что хотите, AI делает. Быстро, но без гарантий.
- **Агентский режим** — AI работает по вашим правилам, вы одобряете каждое действие. Медленнее, но надёжнее.
- **Мультиагенты** — несколько AI-агентов работают как команда: один планирует, другой пишет, третий проверяет.

### 5.2. В чём разница

| | Вайбкодинг | Агентский режим |
|---|---|---|
| Нужно ли понимать код | Нет | Желательно |
| Контроль над результатом | Минимальный | Полный |
| Как проверяете | «Работает — ок» | Проверяете каждое изменение |
| Тестирование | Руками «потыкали» | Автоматические тесты |
| Повторяемость результата | Низкая | Высокая |
| Для серьёзных проектов | Рискованно | Да, с правильной настройкой |

### 5.3. Как повысить качество вайбкодинга

Вайбкодинг не обязан быть хаотичным. Несколько простых практик позволяют сохранить скорость, но значительно повысить качество результата.

#### Сначала опиши — потом делай (Spec-Driven Development)

Самый мощный приём: не просить AI сразу создавать программу. Вместо этого — попросить его сначала написать подробное описание (спецификацию) того, что будет создано. Проверить описание. И только потом дать команду делать.

```
# Запрос 1 (описание, НЕ код)
"Мне нужно приложение для учёта рабочего времени.
НЕ создавай его пока. Сначала опиши:
- Какие данные будут храниться
- Какие экраны будет видеть пользователь
- Какие действия доступны
- Какие есть ограничения и правила
- Какие технологии ты предлагаешь и почему"

# Запрос 2 (правки к описанию)
"Добавь: таймер должен продолжать работать, даже если
закрыть вкладку браузера. И убери экран настроек —
он не нужен на первом этапе."

# Запрос 3 (теперь создавай)
"Отлично. Теперь создай приложение строго по этому описанию."
```

**Почему это работает:** AI получает полную и чёткую картину, вместо того чтобы угадывать из одного предложения. Результат гораздо предсказуемее.

#### Сохраняйте удачные версии (Git)

Git — это система, которая хранит историю всех изменений в проекте. Как «отмена» в текстовом редакторе, но для всего проекта целиком.

Зачем это нужно: следующий запрос к AI может сломать то, что уже работало. Если вы сохранили предыдущую версию — всегда можно вернуться.

```bash
# Перед началом работы
git init && git add -A && git commit -m "начало проекта"

# После каждого удачного изменения
git add -A && git commit -m "добавил фильтрацию"

# Что-то сломалось? Возвращаем как было
git checkout -- .
```

**Правило:** AI сделал что-то хорошее → сохраните. Особенно в вайбкодинге, где каждый следующий запрос — это лотерея.

#### Никогда не передавайте AI пароли и секретные ключи

AI-инструменты сохраняют историю ваших запросов. Некоторые отправляют текст на свои серверы. Если вы вставите пароль от базы данных или секретный ключ от сервиса — это утечка.

```
# ✗ НИКОГДА так
"Подключись к базе данных: пароль admin123, адрес prod-db.company.com"
"Используй ключ: sk-proj-abc123xyz..."

# ✓ Правильно
"Подключись к базе данных. Адрес и пароль возьми
из переменных окружения (DATABASE_URL)."
```

*(Переменные окружения — это способ хранить секреты отдельно от кода, в специальном защищённом файле)*

**Что делать:**
- Храните пароли и ключи в отдельном файле `.env`, который не попадает в общий доступ
- Если секрет случайно попал в код — считайте его скомпрометированным и меняйте немедленно

#### Просите AI проверять свою работу

Даже в вайбкодинге полезно попросить AI самого проверить результат:

```
"Проверь, что приложение:
1. Запускается без ошибок
2. Основной сценарий работает (создать → сохранить → увидеть)
3. Всё, что работало раньше, не сломалось
4. На мобильном телефоне выглядит нормально"
```

Ещё лучше — просить AI писать автоматические проверки прямо в процессе:

```
"Ты добавил фильтрацию по дате. Теперь напиши 3 проверки:
фильтр по одной дате, фильтр по диапазону дат,
и что происходит, когда ничего не найдено."
```

### 5.4. Путь: от прототипа к настоящему продукту

Лучшая стратегия — использовать вайбкодинг как первый шаг, а не единственный:

```
1. Вайбкодинг       → Быстрый прототип, проверка идеи
2. Оценка           → Берём в настоящую разработку или идея не взлетела?
3. Агентский режим  → Создаём заново, уже с правилами и стандартами,
                       используя прототип как образец
4. Стандартизация   → Фиксируем правила для команды
```

> *Прототип — это не код для повторного использования. Это наглядное техническое задание, записанное языком работающего приложения.*

## 6. Практические рекомендации

### 6.1. Когда стоит вайбкодить — чеклист

- ✓ Прототип или демо
- ✓ Внутренний инструмент для команды
- ✓ Одноразовая задача (скрипт, отчёт, конвертация)
- ✓ Эксперимент с новой технологией
- ✓ Посадочная страница или маркетинговый сайт
- ✓ Хакатон

### 6.2. Как получить лучший результат

- **Описывайте конечный результат**, а не шаги: «Дашборд с тремя графиками» вместо технических подробностей
- **Давайте контекст**: для кого это, какой стиль, сколько данных
- **Двигайтесь мелкими шагами**: один запрос — одна функция
- **Указывайте технологии**, если они важны для вас. И не указывайте, если вам без разницы — дайте AI выбрать оптимальное
- **Сохраняйте удачные версии**: если текущий результат нравится — зафиксируйте его перед следующим запросом

### 6.3. Полезные приёмы общения с AI

#### Базовые шаблоны запросов

```
# Начало нового проекта
"Создай [что нужно]. Основные функции: [1], [2], [3].
Дизайн: [стиль]. Тема: тёмная/светлая."

# Доработка
"В текущем приложении не хватает [чего].
Добавь [описание]. Не трогай [что не менять]."

# Исправление проблемы
"Когда я [делаю действие], происходит [проблема].
Должно быть: [как правильно]. Покажи что изменил."
```

#### Brainstorm: AI как соавтор идеи

Вместо того чтобы сразу просить результат, поговорите с AI. Обсудите задачу. Это сильно улучшает итог.

```
# Генерация вариантов
"Мне нужно приложение для [задача]. НЕ делай его пока.
Предложи 3 варианта с разным уровнем сложности:
- Простой (за час)
- Средний (полноценный прототип)
- Продвинутый (готов к показу клиенту)
Для каждого: что будет уметь и сколько примерно времени займёт."

# Уточнение требований
"Я хочу сделать [описание]. Задай мне 5 вопросов,
которые помогут уточнить, что именно нужно."

# Сравнение подходов
"Мне нужна авторизация (вход пользователей) в приложении.
Какие есть варианты? Сравни их: что проще, что безопаснее,
что быстрее реализовать."
```

**Почему это работает:** AI отлично генерирует варианты и задаёт уточняющие вопросы. 5 минут обсуждения экономят час переделок.

#### Superprompt: всё в одном запросе

Чем больше деталей вы дадите в одном сообщении — тем точнее будет результат и тем меньше придётся потом доделывать.

```
"Создай веб-приложение для трекинга привычек.

Для кого: для меня лично, буду открывать с телефона.

Что должно уметь:
- Список привычек с галочками на каждый день
- Вид по неделям (понедельник — воскресенье)
- Счётчик серии (сколько дней подряд выполняется)
- Кнопка «добавить привычку» с названием и эмодзи
- Данные сохраняются в браузере (без регистрации)

Как должно выглядеть:
- Тёмный фон, скруглённые карточки
- Зелёный цвет для выполненных дней
- Нормально выглядит на экране телефона

НЕ нужно:
- Вход / регистрация
- Серверная часть
- Статистика и графики"
```

#### Попросите AI назначить себе роль

Простой приём, который заметно улучшает качество: попросите AI «стать» экспертом.

```
"Представь, что ты опытный дизайнер интерфейсов.

Задача: создай страницу с таблицей, которую можно
сортировать и фильтровать.

Важно:
- Таблица должна быть удобной на мобильном
- Фильтры не должны занимать много места
- Данные для таблицы я передам потом, не придумывай сам"
```

#### После создания: попросите улучшить

Когда первая версия готова — попросите AI самого её покритиковать и улучшить:

```
# Самопроверка
"Посмотри на то, что ты создал, критически.
Найди 3 самых слабых места и предложи как их исправить."

# Граничные случаи
"Что будет, если: данных нет совсем? Данных очень много?
Пользователь ввёл что-то неожиданное?"

# Финальная полировка
"Добавь обработку ошибок — чтобы приложение
не падало, а показывало понятное сообщение."
```

### 6.4. Что попробовать прямо сейчас

1. Откройте **bolt.new** или **v0.dev** в браузере
2. Опишите простой инструмент, который был бы полезен вашей команде
3. Поговорите с AI 5–10 минут, улучшая результат
4. Оцените: можно ли этим пользоваться уже сейчас?
